# T3 — Backend / Simulation Engine

> **Role:** Central brain — device simulation, Ollama LLM analysis, rule engine, WebSocket streaming
> **Port:** 5174
> **Stack:** Python 3.11+ / FastAPI + WebSocket + Ollama + Rule Engine + Device Simulator
> **Connects to:** T1 (commands in), T2 (alerts/solutions out)

## Core Components

### 1. Device Simulator
Simulates 4 MPLS network sites. Each device has:
- Interface state (up/down, utilization)
- BGP session state (established / flapping / down)
- Route table (IPv4 + MPLS labels)
- Latency/throughput metrics
- Power state (online / offline / diagnostic)

**Initial state:** All devices online, BGP established, routes healthy.
**State transitions:** Triggered by T1 commands, T2 actions, or rule engine.

### 2. Ollama LLM Integration
Uses Ollama (offline LLM) for:
- **Root cause analysis** — when an alert fires, ask LLM: "Why is BGP flapping between Bangalore PE1 and Mumbai PE2?" → structured answer (cause + evidence)
- **Solution recommendation** — "What should the operator do to fix this?" → structured steps
- **Confidence scoring** — how confident is the LLM in its analysis (0-100%)
- **Models:** Qwen3-8B (primary), Qwen3-4B-Thinking (fallback)

### 3. Rule Engine
Simple rule-based engine that runs alongside the LLM:
- Threshold rules: "if latency > 200ms → WARNING alert"
- Pattern rules: "if BGP flaps 3 times in 5 minutes → CRITICAL alert"
- State rules: "if device goes offline → CRITICAL alert + recommend diagnostic"
- Rules are configurable via a JSON file
- Rule engine fires alerts to T2 immediately, LLM runs async for deep analysis

### 4. WebSocket Manager
Manages persistent connections to T1 and T2:
- **T1 connections:** Stream device state every 1s, receive command acknowledgements
- **T2 connections:** Stream alerts, solutions, and system health
- Auto-reconnect handling
- Per-client state (each T1 and T2 instance gets its own stream)

### 5. Command Handler
Receives commands from T1 and T2:
- Validates the command (device exists, action is allowed)
- Executes on the device simulator (state transition)
- Returns result (success/failure + response data)
- Logs to command history
- Broadcasts state change to all connected T1/T2 instances

## Anomaly Types (for Trigger Anomaly)

| Type | Effect | T2 Alert |
|------|--------|----------|
| `bgp_flap` | BGP session oscillates up/down | CRITICAL: BGP flapping |
| `latency_spike` | Latency jumps to 300-500ms | WARNING: High latency |
| `packet_loss` | Packet loss climbs to 10-50% | CRITICAL: Packet loss |
| `interface_down` | An interface goes down | CRITICAL: Interface down |
| `route_leak` | Wrong routes appear in table | CRITICAL: Route leak |
| `congestion` | Throughput drops to near zero | WARNING: Congestion |
| `cpu_spike` | CPU utilization jumps to 90%+ | WARNING: High CPU |

## SD-WAN Simulation Details

### IPsec Tunnel Simulation

The IPsec gateway between Bangalore and Delhi is simulated at the state-transition level (no actual StrongSwan inside the container — the device simulator applies IPsec state machine rules):

| State | Description | Transition Trigger |
|-------|-------------|-------------------|
| `INIT` | IKEv2 phase 1 initiating | Tunnel creation |
| `IKE_SA` | IKE security association established | Successful IKE negotiation |
| `CHILD_SA` | IPsec child SA established (ESP tunnel) | Successful IPsec negotiation |
| `ESTABLISHED` | Full tunnel operational, traffic flowing | Both SAs active |
| `REKEYING` | SA lifetime approaching expiry, rekey in progress | Timer-based |
| `DROPPED` | Tunnel down due to timeout or peer failure | DPD timeout, manual trigger |
| `ERROR` | SA mismatch, proposal mismatch, auth failure | Incorrect config |

**Simulated behaviors:**
- Tunnel throughput: 0-1000 Mbps configurable per tunnel
- Rekey interval: configurable (default 3600s), triggers `REKEYING` state
- DPD (Dead Peer Detection): defaults to 10s interval, 3 retries
- NAT-T: enabled by default, wraps ESP in UDP/4500

### GRE Tunnel Simulation

GRE tunnels available as alternative overlay between any two sites:

| Parameter | Range | Default |
|-----------|-------|---------|
| Tunnel MTU | 1400-1500 | 1476 |
| Keepalive interval | 1-60s | 10s |
| Keepalive retries | 1-10 | 3 |
| Key | 0-4294967295 | 0 (none) |
| TOS | 0-255 | Inherit |

### VPN Failover Simulation

When a primary tunnel fails, the backup path activates:

1. Primary IPsec tunnel transitions to `DROPPED`
2. After `failoverDelay` (default 5s), backup tunnel activates
3. BGP sessions over backup tunnel re-establish
4. Traffic shifts within `failoverTime` (default 15s)
5. When primary recovers, traffic re-converges (30s hold-down)

### QoS & Traffic Shaping Simulation

| Feature | Simulation | Trigger Points |
|---------|------------|----------------|
| Traffic shaping | Rate-limit bandwidth per interface | On congestion threshold |
| Priority queuing | High-priority traffic (latency-sensitive) gets priority | DSCP EF (44-46) |
| WRED | Random early drop when queue depth > 50% | Queue depth monitoring |
| Policing | Drop/remark traffic exceeding CIR | Ingress rate > committed rate |
| DSCP marking | Mark packets at trust boundary | Per-interface policy |

**QoS classes simulated:**

| Class | DSCP | Bandwidth Guarantee | Priority |
|-------|------|--------------------|----------|
| Network Control | CS6 (48) | 5% | Highest |
| Voice | EF (46) | 10% | High |
| Video | AF41 (34) | 20% | Medium-High |
| Critical Data | AF31 (26) | 25% | Medium |
| Bulk Data | AF11 (10) | 25% | Medium-Low |
| Best Effort | DF (0) | 15% | Low |

## LLM Model Justification — Qwen3 vs Alternatives

### Comparison Table

| Criterion | **Qwen3-8B** (ours) | Mistral 7B | LLaMA 3.1-8B | Phi-3-Medium (14B) |
|-----------|---------------------|------------|---------------|-------------------|
| **Context window** | **128K tokens** | 32K | 128K | 128K |
| **VRAM requirement** | **~5.5 GB** (Q4) | ~4.5 GB | ~5.5 GB | ~8.5 GB (won't fit with other services) |
| **Inference speed on RTX 4060** | **~35 tok/s** (Q4) | ~40 tok/s | ~30 tok/s | ~15 tok/s |
| **Reasoning quality (network domain)** | **High** (trained on technical code + math) | Medium | High | Medium-High |
| **Structured output (JSON)** | **Good** (follows system prompt well) | Good | Excellent | Good |
| **Hindi/English code-switch** | **Native support** (trained on bilingual data) | English only | English only | English only |
| **Function calling** | **Good** (tool-use trained) | Limited | Good | Limited |
| **RAG compatibility** | **Excellent** (128K context handles full runbook) | Good | Excellent | Excellent |
| **ONNX / GGUF support** | **GGUF available** (llama.cpp compatible) | GGUF available | GGUF available | GGUF available |
| **Community & updates** | **Active** (Alibaba, frequent updates) | Very active | Very active | Moderate |

### Why Qwen3-8B Won

| Reason | Detail |
|--------|--------|
| **128K context** | A full runbook (~4K tokens) + 5 retrieved chunks (~10K) + telemetry data (~8K) + conversation history (~10K) = ~32K. Qwen3's 128K leaves margin; Mistral's 32K is tight |
| **Bilingual capability** | PS13 is for an Indian NOC. Qwen3's Hindi/English support means operators can query in Hindi and get Hindi responses — Mistral/LLaMA/Phi-3 are English-only |
| **RTX 4060 fit** | Qwen3-8B at Q4 fits in ~5.5GB, leaving 2.5GB for ChromaDB, models, and system. Phi-3-Medium at 14B needs >8.5GB — won't fit on an 8GB card |
| **Network domain reasoning** | Qwen3-8B is trained on a large corpus of technical code and STEM, giving it strong structured reasoning (Q1=What, Q2=Why, Q3=How) — critical for diagnostic accuracy |
| **Ecosystem maturity** | GGUF quantization, Ollama integration, and ONNX export all proven for Qwen3 series |

### Fallback Strategy

| Model | When Used | VRAM | Tokens/sec |
|-------|-----------|------|------------|
| **Qwen3-8B** (default) | Full analysis, root cause, solution recommendations | ~5.5 GB | ~35 tok/s |
| **Qwen3-4B-Thinking** (fallback) | When VRAM < 4GB available, rapid/lightweight queries | ~2.5 GB | ~55 tok/s |
| **Rule engine only** (no LLM) | When Ollama is unavailable or VRAM fully consumed | 0 GB | Instant |

### Quantization Strategy

| Level | Bits | VRAM | Quality Retention | Use Case |
|-------|------|------|-------------------|----------|
| Q4_K_M | 4-bit | ~5.5 GB | ~95% | Default for daily ops |
| Q5_K_M | 5-bit | ~6.5 GB | ~97% | When full accuracy needed (post-incident analysis) |
| Q8_0 | 8-bit | ~8.0 GB | ~99% | Critical analysis only (may cause OOM with other services) |

## Running the Engine

```bash
# Start T3
cd t3/
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --port 5174 --reload

# Ensure Ollama is running (separate process)
ollama serve
ollama pull qwen3:8b
ollama pull qwen3:4b-thinking
```
