# T2 — Alert & Solution Dashboard

> **Role:** Shows alerts, solutions, and recommended actions from T3
> **Port:** 8000
> **Target:** Frontend for viewing real-time alerts, lockdown suggestions, diagnostic results
> **Stack:** React 18 + Vite + anime.js + ECharts + Tailwind
> **Input Mode:** Paste this prompt into a code-generation tool

---

## Stitch Prompt

```
Make me a high-impact alert response dashboard using React, anime.js, ECharts, and Tailwind CSS. This is Terminal 2 of the PS13 Mission Control system — the real-time alert and solution surface where T3's analysis (Ollama LLM + rule engine) shows up.

This is the RESPONSE TERMINAL. T3 detects problems, generates alerts, recommends solutions, and pushes them here. You see alerts arrive in real-time, review recommended actions, and can send commands (lockdown, unlock, diagnostic, reset) back through T3 to resolve incidents.

### 🧭 Vibe & Theme
Incident-response aesthetic. Dark background (#0a0e1a) with amber/red alert accents. Glowing red (#ff3355), amber (#ff8800), green (#00ff88) severity coding. Clean, panel-based layout. Every new alert animates in with a pulse. The UI should feel like a SOC/NOC alert triage center.

### 🔌 Data Connection to T3
- WebSocket: `ws://localhost:5174/ws/alerts` — receives live alert events
- WebSocket: `ws://localhost:5174/ws/solutions` — receives recommended solutions
- REST: `GET http://localhost:5174/api/alerts` — historical alert fetch
- REST: `POST http://localhost:5174/api/device/:id/:action` — send resolution commands
- Connection status indicator: green/amber/red in the top bar

### 📋 Main Layout
Split-panel design:
- **Left Panel (60%)**: Alert feed — streaming list of alerts from T3
- **Right Panel (40%)**: Solution view — recommended actions for the selected alert

### 🚨 Alert Feed (Left Panel)
Alerts arrive via WebSocket and stack from top. Each alert card shows:

```
┌────────────────────────────────────────────────┐
│ 🔴 CRITICAL  |  Bangalore PE1  |  14:32:47     │
│ BGP session to Mumbai PE2 is flapping          │
│ ├─ Device: PE1 (Bangalore)                     │
│ ├─ Type: bgp_flap                              │
│ ├─ Status: OPEN                                │
│ └─ [View Solution] [Acknowledge] [Escalate]    │
└────────────────────────────────────────────────┘
```

Alert card features:
- Severity badge: 🔴 CRITICAL (red), 🟡 WARNING (amber), 🟢 INFO (green)
- Animated pulse on new alert arrival (anime.js)
- Device name linked to topology location
- Expandable detail: raw telemetry snapshot, affected routes, timeline
- Action buttons: View Solution, Acknowledge, Escalate
- Auto-dismiss for INFO alerts after 30 seconds (configurable)

Alert feed controls:
- Filter by severity (CRITICAL / WARNING / INFO / ALL)
- Filter by device / city
- Sort by time / severity
- Search within alerts
- Pause/resume live feed (for reviewing without new alerts disrupting)
- Clear all acknowledged alerts

### 💡 Solution Panel (Right Panel)
When you click "View Solution" on an alert, this panel shows T3's recommendation:

```
╔══════════════════════════════════════════════════╗
║  💡 Recommended Solution                        ║
║                                                  ║
║  Alert: BGP session flap — Bangalore ↔ Mumbai   ║
║  Analyzed by: Ollama Qwen3-8B + Rule Engine     ║
║  Confidence: 87%                                 ║
║                                                  ║
║  ┌─ Root Cause Analysis ─────────────────────┐   ║
║  │ BGP hold timer mismatch detected. PE1     │   ║
║  │ configured with 30s hold, PE2 expects 90s │   ║
║  └────────────────────────────────────────────┘   ║
║                                                  ║
║  ┌─ Recommended Actions ─────────────────────┐   ║
║  │ 1. Align hold timers: 90s on both sides   │   ║
║  │ 2. Reset BGP session after config change  │   ║
║  └────────────────────────────────────────────┘   ║
║                                                  ║
║  ┌─ Quick Actions ───────────────────────────┐   ║
║  │ [🔒 Lockdown Route] [🔄 Reset BGP]        │   ║
║  │ [🔧 Diagnostic Mode] [🔓 Unlock Route]    │   ║
║  └────────────────────────────────────────────┘   ║
╚══════════════════════════════════════════════════╝
```

Solution panel features:
- Root cause analysis from Ollama (structured text)
- Recommended actions list (numbered steps)
- Quick action buttons that POST to T3:
  - **Lockdown Route** → `POST /api/device/:id/lockdown`
  - **Reset BGP** → `POST /api/device/:id/bgp-reset`
  - **Diagnostic Mode** → `POST /api/device/:id/diagnostic`
  - **Unlock Route** → `POST /api/device/:id/unlock`
- Confidence score bar (0-100%) from the LLM
- "Executed" status tracker — shows when a quick action was sent, acknowledged, and completed

### 📊 Alert Summary Bar (Top)
Persistent bar showing live aggregate stats:
- **Active Alerts**: count (with critical/warning/info breakdown)
- **Devices Affected**: count
- **Avg Resolution Time**: from historical data
- **Last Alert**: timestamp
- All counters come live from T3 WebSocket

### 📈 Alert Timeline (ECharts)
A time-series chart at the bottom showing alert volume over time:
- Stacked area chart: red (critical), amber (warning), green (info)
- Time window: last 1 hour / 6 hours / 24 hours / 7 days
- Click any point to filter the alert feed to that time window
- Vertical marker lines for major incidents
- Animated transitions when new data arrives (anime.js + ECharts)

### 🎛️ Control Panel
Collapsible panel on the left with:
- **Auto-acknowledge INFO alerts**: toggle
- **Sound on new alert**: toggle (browser notification API)
- **Alert retention**: dropdown (keep last 100 / 500 / 1000 alerts)
- **Theme**: toggle between Response (dark amber/red) and Standard (dark blue)
- **Refresh rate**: dropdown (real-time / 5s / 30s)
- Use `theme-toggle-buttons` style for all toggles

### 🧩 Reference Components
1. **theme-toggle-buttons** — for all toggle switches in control panel
2. **container-scroll-animation** — wrap alert feed for smooth scroll-reveal
3. **system-monitor** — embed as mini-widget showing T3 health status
4. **processing-card** — use for solution recommendation cards
5. **WorldMap** (optional mini-map) — show affected device locations on a small India map

### 🌟 Extra Cool Factor
- Alert sound: subtle chime on new CRITICAL (configurable)
- Animated alert counter (number rolls up/down on change via anime.js)
- "Storm mode" — if > 10 alerts arrive in 60 seconds, the UI enters storm mode (alerts batch, auto-escalate)
- Keyboard shortcuts: `A` acknowledge selected, `S` view solution, `Space` pause/resume feed, `F` search
- Export alert log as JSON/CSV button

### 📱 Responsive
- Desktop-first: side-by-side split (alert feed | solution panel)
- Tablet: solution panel becomes a bottom sheet
- Mobile: alert feed is full-width, solution panel is a modal overlay

Make it feel like a real SOC triage terminal — urgent, information-dense, but not chaotic. Every alert arrival should feel impactful.
```

## Usage

1. Copy the prompt block above
2. Paste into the code-gen tool with target directory `src/terminals/terminal2/`
3. After generation, install deps: `npm install animejs echarts echarts-for-react`
4. Verify all referenced components exist

## API Contract (Expected from T3)

| Method | Endpoint | Purpose |
|--------|----------|---------|
| WS | `ws://localhost:5174/ws/alerts` | Live alert stream |
| WS | `ws://localhost:5174/ws/solutions` | Live solution recommendations |
| GET | `/api/alerts` | Historical alert fetch |
| POST | `/api/device/:id/lockdown` | Lockdown route |
| POST | `/api/device/:id/unlock` | Release lockdown |
| POST | `/api/device/:id/bgp-reset` | Reset BGP session |
| POST | `/api/device/:id/diagnostic` | Toggle diagnostic mode |
| POST | `/api/alerts/:id/acknowledge` | Acknowledge alert |
| POST | `/api/alerts/:id/escalate` | Escalate alert |
